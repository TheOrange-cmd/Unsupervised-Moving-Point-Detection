# M-Detector Refactoring (ROS-Independent)

## Project Description

The goal of this project is to refactor the existing m-detector code, originally a ROS C++ project for detecting moving events in LiDAR data. The refactored version aims to operate without ROS dependencies and provide a convenient Python interface for offline data processing.

The core C++ algorithm will be preserved, while Python libraries (like the nuscenes-devkit) will be used for data preparation. The final system should support multiple datasets (e.g., nuScenes, TruckSim, Waymo, KITTI) in a way that the C++ core remains dataset-agnostic. Data preparation in Python will ensure consistency, feeding a standardized format into the C++ algorithm, although configuration files can be tuned per dataset.

**Note on Odometry:** The original m-detector relied on odometry generated by `fast-lio2` (another ROS package) using LiDAR and raw IMU data. Since datasets like nuScenes provide accurate ego-pose information, this project aims to use that directly, potentially eliminating the need for `fast-lio2`. This significantly reduces refactoring effort. Further investigation will compare `fast-lio2` odometry with nuScenes ego-pose data to determine if any differences need accommodation in the refactored code.

## Refactoring Approach

The refactoring process is planned in the following phases:

### Phase 1: Isolate and Test DynObjFilter without ROS

1.  **Setup & Baseline:**
    *   Create a new branch or copy of the project for refactoring.
    *   Ensure the original ROS version builds and runs correctly (establish baseline).
    *   Set up a new CMake project (`CMakeLists.txt`) without catkin or ROS dependencies. It should find Eigen, PCL (if needed by `DynObjFilter`), and require C++17 (for `std::execution::par`).

2.  **Remove ROS Includes & Basic Types (within `DynObjFilter` files):**
    *   Target files: `DynObjFilter.h`, `DynObjFilter.cpp`, and related helpers (`RingBuffer`, `DepthMap`, `point_soph`).
    *   Remove `#include <ros/ros.h>` and other ROS headers.
    *   Replace `ros::Time` with `std::chrono::time_point<std::chrono::high_resolution_clock>` or `double` for timestamps consistently.
    *   Replace ROS logging (`ROS_INFO`, etc.) with `std::cout` or a logging library (e.g., spdlog).
    *   **Goal:** Compile `DynObjFilter.cpp` and its direct non-ROS dependencies using the new CMake setup (linking/running not expected yet).

3.  **Create a Standalone `main` Entry Point:**
    *   Create `main.cpp` (or similar).
    *   Include `DynObjFilter.h`.
    *   Instantiate `DynObjFilter`: `std::shared_ptr<DynObjFilter> dyn_filter(new DynObjFilter());`.
    *   **Goal:** Create a basic executable structure including the filter class.

4.  **Replace ROS Parameter Loading:**
    *   Modify `DynObjFilter::init(ros::NodeHandle& nh)` to accept parameters differently (e.g., via a struct, config file path, or arguments).
    *   Implement a simple configuration loader in `main.cpp` (e.g., using nlohmann/json or yaml-cpp) to read parameters needed by `DynObjFilter`.
    *   Call the modified `dyn_filter->init(...)` with loaded parameters.
    *   **Goal:** Initialize `DynObjFilter` without `ros::NodeHandle`.

5.  **Replace ROS Callbacks with Direct Calls (Mock Data):**
    *   Original: `PointsCallback`, `OdomCallback` triggered `DynObjFilter::filter` via a timer.
    *   New `main.cpp`: Create a simple loop.
    *   Inside loop, create mock input data:
        *   `pcl::PointCloud<PointType>::Ptr feats_undistort` (populate with dummy points).
        *   `Eigen::Matrix3d rot_end` (e.g., identity).
        *   `Eigen::Vector3d pos_end` (e.g., zero).
        *   `double scan_end_time` (e.g., incrementing).
    *   Directly call `dyn_filter->filter(feats_undistort, rot_end, pos_end, scan_end_time);`.
    *   **Goal:** Run core filtering logic on demand with controlled inputs, independent of ROS topics/callbacks. Test for crashes.

6.  **Replace ROS Output with Simple Verification:**
    *   Original: `DynObjFilter::filter` populated member point clouds (`laserCloudDynObj_world`, `laserCloudSteadObj`), `publish_dyn` used ROS publishers.
    *   Remove or modify `publish_dyn` to eliminate ROS publishers.
    *   After calling `dyn_filter->filter`, access resulting point clouds (`dyn_filter->laserCloudDynObj_world`, `dyn_filter->laserCloudSteadObj`).
    *   Add simple checks: Print point counts (dynamic/static). Optionally save clouds to PCD files (`pcl::io::savePCDFileASCII`) for visual inspection.
    *   **Goal:** Verify filter output based on mock input without ROS publishers.

### Phase 2: Integrate nuScenes Data Loading

1.  **nuScenes Data Input Pipeline:**
    *   Develop Python script(s) using `nuscenes-devkit` to iterate through scenes/sweeps, load LiDAR paths, and ego poses.
    *   Use `pybind11` for communication between Python and C++.
    *   Feed loaded point cloud and ego pose (`rot_end`, `pos_end`, `scan_end_time`) into `dyn_filter->filter` within the main loop.
    *   **Goal:** Run the standalone filter on actual nuScenes data. Test progressively: single sweep -> single scene -> full dataset.

2.  **Handling `ego_pose`:**
    *   The `DynObjFilter::filter` signature (`const M3D & rot_end, const V3D & pos_end, const double & scan_end_time`) directly maps to nuScenes `ego_pose.json` data for corresponding LiDAR `sample_data` timestamps.
    *   **Goal:** Confirm correct extraction and passing of `ego_pose` data to the filter function.

### Phase 3: Refinements and Future Work Prep

1.  **Testing and Debugging:**
    *   Thoroughly test output against the baseline ROS version (if feasible) or via visual inspection of saved PCD files. Debug discrepancies.

2.  **Offline Processing (Past/Future):**
    *   The existing `RingBuffer` (`buffer`) and `DepthMap` history (`depth_map_list`) are suitable for offline processing.
    *   Loading sequential offline data allows easy access to multiple past/future scans/poses.
    *   Potential modifications for past/future tracking:
        *   Increase size/duration managed by `RingBuffer` and `depth_map_list`.
        *   Adjust logic in `Case1/2/3` or add new logic to query future/past data explicitly.
    *   The refactored structure facilitates these algorithmic changes without ROS constraints.

3.  **Re-integrate Clustering/Frame-Out Processing (Later):**
    *   Once `DynObjFilter` is stable, apply similar refactoring steps (remove ROS, replace params/callbacks/publishers) to `DynObjCluster`.
    *   Call clustering after `DynObjFilter::filter`, passing necessary data.
    *   Alternatively, implement new frame-out processing to enhance temporal/spatial consistency of point-based detections, potentially using past/future data.

4.  **Dataset Expansion:**
    *   Investigate processing other datasets (TruckSim, Waymo, KITTI, etc.) similarly.
    *   Aim for a setup where adding datasets involves writing Python data loaders feeding the existing `pybind11` interface, without C++ code changes.

5.  **Further Optimizations:**
    *   Investigate other potential optimizations or dependency removals identified during refactoring. Address these after achieving the primary goals (ROS removal, functional comparison) to avoid complicating debugging.

## Development Environment Setup

Follow these steps to set up the Conda environment for building and running the project.

### 1. Install Miniconda

If you don't have Conda installed:

```bash
# Download the installer
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh

# Run the installer
bash Miniconda3-latest-Linux-x86_64.sh

# Follow prompts - say 'yes' to initializing Conda when asked.
# Close and reopen your terminal or source your shell configuration:
source ~/.bashrc # Or ~/.zshrc, etc.
```

### 2. Configure Conda Channels

Ensure conda-forge is prioritized:

```
conda config --add channels conda-forge
conda config --set channel_priority strict
```

### 3. Create Conda Environment

Create and activate the environment:
```
conda create -n mdet_env python=3.9
conda activate mdet_env
```

### 4. Install Dependencies

Install necessary build tools, C++ libraries, and Python libraries:
```
# Build tools
conda install cmake make gcc_linux-64 gxx_linux-64

# C++ Libraries
conda install eigen pcl opencv tbb yaml-cpp

# Graphics/Display related (might be needed by PCL/OpenCV visualization)
conda install libglu xorg-libx11 xorg-libxext xorg-libxfixes mesa-libgl-devel-cos7-x86_64 mesa-libgl-cos7-x86_64

# Python binding library and a crypto dependency (sometimes needed indirectly)
conda install pybind11 libxcrypt
```

### Building and Testing

To clean the build directory, configure using CMake, build the project, run tests, and then rerun only the failed tests with verbose output:
```
# Ensure the environment is active
conda activate mdet_env

# Navigate to your build directory (assuming it's inside the project root)
cd path/to/your/Unsupervised-Moving-Point-Detection/build # Create 'build' if it doesn't exist

# Run the build and test sequence - be careful that you are working in the build directory
rm -rf * && cmake .. && make && ctest && ctest --rerun-failed -V

# Or to only run specific tests:
rm -rf * && cmake .. && make -j32 && ctest -V -L filtering
```

# Setting Up Consistent C++ Formatting with clang-format and VS Code

This guide explains how to ensure consistent C++ code formatting across the team using `clang-format` built from source and a shared VS Code task configuration.

**Goal:** Every team member can format the currently open C++ file using a VS Code task (or keyboard shortcut) that relies on a locally built `clang-format`, without needing user-specific paths in the shared configuration.

## 1. Build clang-format from Source

We will build `clang-format` as part of the Clang toolset within the LLVM project.

1.  **Clone the LLVM Project:**
    Choose a location to download the source code (e.g., `~/llvm-project`).
    ```bash
    cd ~
    git clone https://github.com/llvm/llvm-project.git
    cd llvm-project
    ```

2.  **Create a Build Directory:** It's best practice to build outside the source directory.
    ```bash
    mkdir build
    cd build
    ```

3.  **Configure the Build with CMake:**
    This command tells CMake how to prepare the build. We only enable the `clang` project (which includes `clang-format`) and specify an installation directory *relative to the build directory*.

        ```bash
        cmake ../llvm \
              -DCMAKE_BUILD_TYPE=Release \
              -DLLVM_ENABLE_PROJECTS=clang \
              -DCMAKE_INSTALL_PREFIX=../install
        ```
    *   *Explanation:*
        *   `../llvm`: Path to the LLVM source code (CMakeLists.txt).
        *   `CMAKE_BUILD_TYPE=Release`: Builds optimized binaries.
        *   `LLVM_ENABLE_PROJECTS=clang`: Only build the Clang project (saves significant time).
        *   `CMAKE_INSTALL_PREFIX=../install`: Sets a local install directory (`~/llvm-project/install`).

4.  **Build clang-format:**
        ```bash
        make clang-format -j$4 # Uses four CPU cores - consider if more cores is acceptable as this process can take a while
        ```

5.  **Install Locally:** This copies the necessary built files to the `install` directory we specified.
    *   **If using Make:**
        ```bash
        make install
        ```
    After this step, you should find the `clang-format` binary inside `~/llvm-project/install/bin/`.

## 3. Install clang-format to Conventional Location

For the shared VS Code task to work, every team member needs to place the `clang-format` binary in the *same relative path* within their home directory. We will use `~/.clang-format/bin/`.

1.  **Create the Target Directory:**
    ```bash
    mkdir -p ~/.clang-format/bin
    ```

2.  **Copy the Binary:** Copy the built `clang-format` from the local install directory to the conventional location.
    ```bash
    cp ~/llvm-project/install/bin/clang-format ~/.clang-format/bin/
    ```

3.  **Verify Execution:** Make sure it runs.
    ```bash
    ~/.clang-format/bin/clang-format --version
    ```

## 4. Configure VS Code Task

1.  **Create/Open `tasks.json`:**
    *   In VS Code, open the Command Palette (`Ctrl+Shift+P` or `Cmd+Shift+P`).
    *   Type `Tasks: Configure Task`.
    *   If prompted, select "Create tasks.json file from template", then choose "Others".
    *   This creates/opens `.vscode/tasks.json` in your project root.

2.  **Add the Formatting Task:** Replace the contents of `tasks.json` with this:

    ```json
    {
        "version": "2.0.0",
        "tasks": [
            {
                "label": "Format Current File (clang-format)", // Name shown in VS Code
                "type": "shell",
                // Use ${userHome} variable for portability
                "command": "${userHome}/.clang-format/bin/clang-format",
                "args": [
                    "-i",                                // Format in-place (overwrites file!)
                    "--style=Google",                    // Or your team's chosen style (e.g., LLVM, File)
                    "${file}"                            // VS Code variable for the current file path
                ],
                "problemMatcher": [],
                "presentation": {
                    "reveal": "never",                   // Don't show the terminal panel
                    "panel": "shared",
                    "showReuseMessage": false,
                    "clear": false
                },
                "group": {
                    "kind": "build",
                    "isDefault": false
                }
            }
        ]
    }
    ```
    *   **Key Points:**
        *   `command`: Uses `${userHome}` which automatically resolves to the user's home directory (e.g., `/home/drugge`). This requires the convention from Step 3.
        *   `args`:
            *   `-i`: **Formats the file in-place.** Ensure you use version control (Git) and commit changes before formatting, especially when first setting up.
            *   `--style=Google`: Change this if your team uses a different base style or a custom `.clang-format` file (see below).
            *   `${file}`: Passes the currently active file to the command.

3.  **Commit `.vscode/tasks.json`:** Add this file to your Git repository so the whole team gets the task definition.

## 5. Usage

1.  **Open a C++ File:** Open any `.cpp`, `.h`, or `.hpp` file you want to format.
2.  **Run the Task:**
    *   Open the Command Palette (`Ctrl+Shift+P`).
    *   Type `Tasks: Run Task`.
    *   Select "Format Current File (clang-format)".
    *   The file should be formatted according to the specified style.

3.  **(Optional) Keyboard Shortcut:** For faster access:
    *   Open the Command Palette (`Ctrl+Shift+P`).
    *   Type `Preferences: Open Keyboard Shortcuts (JSON)`.
    *   Add this entry to the JSON list (adjust `key` as desired):
        ```json
        {
            "key": "ctrl+alt+f", // Example shortcut
            "command": "workbench.action.tasks.runTask",
            "args": "Format Current File (clang-format)" // Must match the task "label"
        }
        ```
    *   Save the file. Now `Ctrl+Alt+F` (or your shortcut) will trigger the formatting task.
